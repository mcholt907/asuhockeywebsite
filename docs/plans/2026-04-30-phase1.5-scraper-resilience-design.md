# Phase 1.5 Scraper Resilience Design

**Date:** 2026-04-30
**Status:** Draft
**Source:** Production deploy investigation after Phase 1; follow-up to commits `2e1bc4e` (alumni fallback), `b1d6781` (CSP), `9b8c3e6` (puppeteer cache).

---

## Context

After Phase 1 deployed, Render builds started timing out on prerender because EliteProspects returned 403 for every alumni-page request. The other agent's session shipped a bundled `data/asu_alumni_fallback.json` with `ALUMNI_SCRAPE_LIVE=true` as the override flag. That stopped the bleeding.

**The 403s are an IP-range block, not a behavior issue.** Three signals:

1. The exact same code works from a residential IP locally and 403s from Render.
2. The Puppeteer fallback in [utils/request-helper.js:39–58](utils/request-helper.js#L39) — a real Chrome with full JS execution — also gets 403'd. Same IP, same block.
3. EliteProspects is on Cloudflare-class bot mitigation, which blocklists known cloud-provider IP ranges by default.

That reframes the architecture decision. The bundled-JSON pattern isn't a workaround; it's the correct production design given the constraint. Recruiting and `manual_news` already work this way — Phase 1.5 propagates the pattern to alumni (already done) and transfers (still vulnerable), and removes the now-useless Puppeteer fallback that's adding latency without rescuing anything.

---

## Goal

Three structural changes that take Render's scraper surface from "fragile" to "predictable," plus an automated local refresh so the bundled JSON doesn't go stale through neglect:

1. **Stop wasting Render cycles on retries that can't succeed.** Gate the Puppeteer fallback off in production; honor 403 as a stop signal with a 24-hour cooldown.
2. **Pre-empt the next failure.** Apply the alumni-style fallback to transfer-scraper *before* its cache TTL expires on Render.
3. **Make the local-refresh workflow boring.** `npm run refresh-data` does the right thing in one command; a Windows Scheduled Task runs it weekly so nobody has to remember.

Out of scope: contacting EliteProspects for permission (worth doing in parallel — see Out of Scope), residential-proxy services (compliance grey area, not pursuing), changing data sources entirely (Phase 3 conversation).

---

## Decisions

### 1. Gate the Puppeteer fallback behind an env flag

[utils/request-helper.js:39–58](utils/request-helper.js#L39) currently launches a full Chrome on every 403 and re-requests the same URL with the same User-Agent (line 47). On Render this:

- Adds 5–10s of cold Chrome startup per blocked URL (alumni does this ~10× per page, ~100× total)
- Cannot succeed because the IP is the trigger
- Costs free-tier RAM that Render doesn't have to spare

**Change:** wrap the Puppeteer block in `if (process.env.SCRAPER_PUPPETEER_FALLBACK === 'true')`. Default is **off**. Local dev that relies on it can set the flag in `.env`. Production never sets it.

The existing `puppeteer` import stays (still used by [scripts/prerender.js](scripts/prerender.js)) — we're gating the *behavior*, not removing the dependency.

### 2. Add 403 cooldown to `request-helper.js`

When an EP 403 is observed, write a marker file (`src/scripts/cache/.403-cooldown-{host}`) with the timestamp. On subsequent requests to the same host, if the marker is fresher than 24h, skip the request entirely and `throw` a sentinel error the scrapers can catch.

Why a marker file rather than in-memory: scrapers run from prerender (separate process) AND the Express server, and Render's free tier restarts daily. In-memory cooldown wouldn't survive any of those. The cache directory IS ephemeral on Render but persists across the same instance's lifetime, which is enough to prevent the "scheduler hammers EP at startup of every request" pattern visible in this morning's logs.

Hostname keyed (not URL keyed): if `eliteprospects.com` blocks one URL it's blocking all of them.

### 3. Bundled fallback for transfers

[transfer-scraper.js](transfer-scraper.js) hits `https://www.eliteprospects.com/team/18066/arizona-state-univ/transfers`. It currently has a 24h cache that's still valid in production from the last successful local refresh, but the next time it tries to revalidate from Render it'll 403 forever.

**Change:** mirror [alumni-scraper.js:16–46](alumni-scraper.js#L16) exactly:

- New `data/asu_transfers_fallback.json` — committed, generated from a local successful scrape.
- New `getFallbackTransferData()` helper.
- Returns the fallback when no valid cache exists, unless `TRANSFER_SCRAPE_LIVE=true`.
- Returns the fallback when scraping throws (e.g. 403), unless `TRANSFER_SCRAPE_LIVE=true`.

### 4. Refresh scripts

Add three npm scripts:

```json
"refresh-alumni":    "ALUMNI_SCRAPE_LIVE=true node scripts/refresh-alumni.js",
"refresh-transfers": "TRANSFER_SCRAPE_LIVE=true node scripts/refresh-transfers.js",
"refresh-data":      "npm run refresh-alumni && npm run refresh-transfers"
```

Each refresh script:

1. Runs the scraper with the live flag.
2. Validates the result (non-empty, expected shape).
3. **If valid**, writes to `data/asu_{topic}_fallback.json`.
4. **If invalid** (empty, scraper threw, etc.), exits non-zero and leaves the existing JSON untouched.

Important: the existing fallback file is **only** overwritten on a successful scrape. A 403'd run from a coffee-shop IP must not zero out good production data.

`scripts/refresh-alumni.js` and `scripts/refresh-transfers.js` are tiny wrappers (~30 lines each) — they exist so the env-flag handling and validation lives in code instead of as inline shell. The scripts also intentionally **do not** call `git add`/`commit`/`push`. That's the next layer.

### 5. Automation via Windows Task Scheduler

**Yes, this is automatable** — but importantly, **only on a residential-IP machine** (your local Windows 11 box). It cannot run on GitHub Actions, Render Cron Jobs, or any cloud scheduler — those would hit the same IP block. The user's machine becomes the de-facto scrape worker.

A new script `scripts/refresh-and-push.cmd` does the full loop:

```cmd
cd /d C:\Users\farkh\asuhockeywebsite
git fetch origin
git checkout main
git pull --ff-only

call npm run refresh-data
if errorlevel 1 (
  echo Refresh failed; exiting without commit. >> .refresh-log.txt
  exit /b 1
)

git diff --quiet -- data\asu_alumni_fallback.json data\asu_transfers_fallback.json
if errorlevel 1 (
  git add data\asu_alumni_fallback.json data\asu_transfers_fallback.json
  git commit -m "data: refresh alumni and transfer fallbacks (automated)"
  git push origin main
  echo %DATE% %TIME% — refreshed and pushed >> .refresh-log.txt
) else (
  echo %DATE% %TIME% — no changes >> .refresh-log.txt
)
```

A companion `scripts/RefreshDataTask.xml` provides a ready-to-import Task Scheduler definition:

- **Trigger**: weekly, Sunday 06:00 local time (you're rarely working then; new alumni updates from EP land mid-week).
- **Action**: run the `.cmd` from the repo root.
- **Settings**: skip if missed (don't run on next startup if the laptop was off — old data is fine, no harm in waiting another week); allow on battery; don't wake the machine.

To install: `schtasks /create /xml scripts\RefreshDataTask.xml /tn "ASU Hockey Data Refresh"`.

**Constraints to be honest about:**

- **Runs only when the laptop is on at the trigger time.** If the laptop is off all week, no refresh — but cache + fallback degrade gracefully. Worst case: the bundled JSON ages by N weeks. Alumni data updates slowly enough that this is fine.
- **Requires git auth to be cached.** You already push from this machine, so that's set.
- **Failures are silent unless you read `.refresh-log.txt`.** A future enhancement is a Sentry breadcrumb or a Windows toast on failure — out of scope for this PR.

If you'd rather not automate and just run `npm run refresh-data` manually before deploys, skip step 5 entirely. The other four changes still hold.

## Verification

1. **Local refresh works with the live flag**: `npm run refresh-data` produces non-empty fallback JSONs.
2. **Refresh is no-op on validation failure**: temporarily simulate a bad scrape (return `[]`) — confirm the existing `data/asu_*_fallback.json` is **not** overwritten and the script exits non-zero.
3. **Production prerender uses fallback**: temporarily delete the local cache directory, run `npm run build`, observe `[Alumni Scraper] Returning bundled fallback alumni data` and `[Transfer Scraper] Returning bundled fallback transfer data` in prerender logs (with `ALUMNI_SCRAPE_LIVE` and `TRANSFER_SCRAPE_LIVE` unset).
4. **Puppeteer fallback off by default**: with `SCRAPER_PUPPETEER_FALLBACK` unset, `requestWithRetry()` against a 403'd URL throws after retries without launching Chrome (verify by stubbing axios in a unit test, or by tailing logs and grepping for "trying Puppeteer fallback" — should never appear on Render).
5. **403 cooldown takes effect**: after one 403 is observed, the next request to the same host within 24h short-circuits without making the network call (assert via Sentry breadcrumb or log line counts).
6. **Scheduled Task fires**: `schtasks /run /tn "ASU Hockey Data Refresh"` executes the `.cmd`; `.refresh-log.txt` shows a "no changes" line within 30s.
7. **Render deploy stays clean**: deploy after this PR; observe alumni + transfer fallback messages in prerender; observe **zero** "trying Puppeteer fallback" lines in production logs.

## Risks & Mitigations

| Risk | Mitigation |
| --- | --- |
| Refresh script accidentally writes empty JSON over good data | Mandatory shape/length validation before the write step (reject if `skaters.length === 0`, etc.); the existing file is only replaced on a fully-validated successful scrape. |
| 403 cooldown marker survives across an EP block being lifted | Cooldown is 24h max; manual override = delete the marker file. |
| Local cache directory missing on Render → fallback path runs every prerender | That's the intended behavior. The fallback path is fast (file read + JSON parse, ~5ms). |
| Scheduled Task fires while you're actively editing the JSON files manually | Diff-check guards against committing your in-progress edits — but two writers in parallel can lose changes. Default trigger is Sunday 06:00 to minimize collision; if needed, change to a less-active window. |
| Local refresh succeeds but git push fails (auth, conflict) | `.cmd` exits non-zero and logs to `.refresh-log.txt`; the data file remains modified locally for next time. No data loss. |
| EliteProspects starts blocking your residential IP too | Then the manual scrape stops working everywhere and the bundled JSON ages. At that point the conversation shifts to alternate data sources — escalate, don't try to evade. |

## Out of Scope (separate work)

- **Contact EliteProspects.** Worth a polite email asking whether automated access is permitted for a non-commercial fan site; include URLs, frequency, and attribution. Low odds of yes, but the email costs nothing and a written answer either way is useful — if granted, this whole design becomes optional.
- **Residential proxy services.** Solves the IP block by paying a service to route requests through residential IPs. Compliance grey area at best, terms-of-service violation at worst. Not recommended.
- **Self-hosted GitHub Actions runner.** Would let CI run the refresh from the user's residential IP. More moving parts than a Windows Scheduled Task and offers no real advantage for a weekly job.
- **Replacing scraping with a paid API.** EliteProspects has a Pro API but pricing is not advertised publicly; revisit if/when this fan site becomes a sustained operation worth paying for.
- **Notifications on refresh failure.** Sentry breadcrumb / Windows toast / email — useful but not load-bearing.
- **Removing `puppeteer` as a dependency.** Still needed by `scripts/prerender.js`. Just gating its scraper-fallback usage.

## Implementation Sequencing

When the implementation doc is written, the four sub-changes can ship in any order, but the safe sequence is:

1. **First commit**: refresh scripts + transfers fallback JSON. Generates the data, doesn't yet rely on it.
2. **Second commit**: transfer-scraper fallback wiring. Now production reads the fallback when needed.
3. **Third commit**: Puppeteer fallback gate + 403 cooldown in `request-helper.js`. Cleans up the noise; production no longer wastes cycles on retries that can't succeed.
4. **Fourth commit**: `refresh-and-push.cmd` + Task Scheduler XML + docs. Automation layer; non-load-bearing.

Each commit is independently revertible; commits 1–3 are the load-bearing ones for production reliability.
